# Work / IT Software - Problem Solving

## 암기 포인트 및 흐름

- 문제 발견 → 바로 보이지 않음 → 로그 확인 → 팀원 도움 → 교훈 순서
- 숫자 하나, 형식 하나 같은 작은 원인을 넣는다
- 급할수록 차근차근 보게 됐다는 결론으로 마무리한다

## 한국어 전문

얼마 전에는 배포 직전에 데이터가 맞지 않는 문제를 발견했습니다.
처음에는 어느 부분이 틀렸는지 바로 보이지 않아서, 로그와 화면을 번갈아 확인했습니다.
30분 정도 지나서 원인을 찾았고, 테스트 데이터를 다시 맞추자 문제가 사라졌습니다.
그때는 조금 답답했지만, 혼자 오래 붙잡지 않고 팀원에게 바로 물어본 것이 도움이 됐습니다.
원인이 작은 숫자 하나였다는 점이 의외였는데, 그래서 더 빨리 찾지 못했던 것 같습니다.
그 경험 이후로는 작은 이상 징후도 그냥 넘기지 않게 됐습니다.
문제가 생기면 서둘러도 되지만, 결국은 차근차근 확인하는 게 가장 빠르다고 느꼈습니다.
그래서 저는 지금도 급할수록 먼저 로그부터 보는 습관을 유지하고 있습니다.

## 영어 전문

A while ago, I found a data mismatch right before a deployment.
At first, I could not see exactly where the problem was, so I kept switching between the logs and the screen.
After about 30 minutes, I found the cause, and the problem disappeared after I fixed the test data.
It was a little frustrating, but asking my teammate early helped me avoid wasting too much time alone.
The surprising part was that the cause was just one small number, which is why it took me longer to find it.
After that, I stopped ignoring small warning signs.
When a problem happens, rushing can help, but I feel careful step-by-step checking is still the fastest way.
So even now, when things feel urgent, I still start with the logs first.

## 키 문장 및 단어

### Key Sentences

- I found a data mismatch before deployment.
- I asked my teammate for help early.
- I always start with the logs when something looks wrong.

### Key Words

- warning sign : 징후
- teammate : 팀원
- urgent : 급한

## 키 한국어 문장 및 대응되는 영어표현 여러개

### 저는 문제가 생기면 팀원에게 바로 물어봅니다.

- I ask my teammate for help right away when something goes wrong.
- I do not wait too long before asking for help.

### 저는 급할수록 로그부터 보는 편입니다.

- I usually start with the logs when something looks wrong.
- I check the logs first when a problem appears.
- I prefer to look at the logs before guessing.

### 원인을 찾으면 수정하고 다시 확인합니다.

- Once I find the cause, I fix it and check again.
- After I find the problem, I make the fix and verify it.
- I usually correct the issue and then test it again.

### 다음에는 같은 문제가 반복되지 않게 메모를 남깁니다.

- I leave a note so the same issue does not repeat later.
- I write it down so we can avoid the same problem next time.
- I keep a short record to prevent the issue from coming back.

## Follow-up Hooks

- What kind of data mismatch was it?
- How did your teammate help?
- Do you usually solve problems alone?
- How do you stay calm when things go wrong?
- What do you check first during debugging?
